An AI OS where every skill is personalized by one knowledge graph

Off-the-shelf assistants are amnesiac — they ask you everything, every time. This is an OS of 42 apps sitting on a markdown knowledge graph of one person's life, so each app is personalized by the facts it reads: your voice, your history, your relationships. Here's the whole system on one page — what's built, the graph underneath, and exactly where the model thinks versus where code decides.

The idea in one picture

Generic skills become personal when they sit on your graph

That amnesia is the problem these apps solve. They sit on top of a knowledge graph of one person's life: people, history, voice, health, finances, all as queryable facts. So the same app code comes out personalized: /forms fills the citizenship application from histories the graph already holds, /draft writes in your voice to this exact person, /friends knows which friendships you're letting slip. Facts are pulled (bronze), parsed into the graph (silver), then projected into the views you read (gold); that pipeline is detailed at the end. What matters first: every app reads one shared graph.

🔁

No per-app memory

Apps don't each keep a private store. They read and write one graph, so a fact one app learns — a friend's new job — is instantly true for every other.

🧬

Personalization is data, not prompts

The same /draft logic sounds like you because your voice profile lives in the graph, not because it was told who you are this session.

The knowledge graph

One entity, many projections

click a node

The graph is the vault — entities are nodes, wikilinks are edges. Priya Kapoor resolves to one canonical row but appears in two gold zones, and her employer edge moved Northwind→Meridian, recorded bi-temporally. Click a node to read its facts.

PersonOrganizationRoleProject

Two orthogonal axes

Every file has a type (entity / note / dashboard / activity-log / spec / archive) and, when it's an entity, an entity_type (Person / Organization / Role / Project / …). Only entities carry current.* + facts[].

Edges are frontmatter links

A wikilink list in frontmatter is the edge — never inline [pred::[[obj]]], which failed to replicate at scale. Queryable in code via silver.query(...).

Bi-temporal facts: try it

Facts are append-only and time-qualified, so the graph answers "what was true then," not just "now." A new fact back-fills the old one's valid_to; current.* is the latest-valid projection.

20222026 (now)
Why it matters

Without bi-temporal facts (Mem0's flagged 2026 failure mode), high-relevance memories go confidently wrong rather than just stale — a "current employer" survives a job change for two regenerations. Graph-DB products ship this natively; this is the markdown-substrate equivalent.

No app edits an entity file directly. Every fact goes through silver.write(), so provenance, resolution, conflict policy, and tier routing are uniform.

There's no privacy_class field — it's computed (the writer's default vs the entity's zone default, more restrictive wins), and the class a fact lands in decides which rails fire and whether it can auto-commit:

The five core halt-not-warn rails: user-only (never shared externally), paraphrase-only (no ≥5-word verbatim third-party quote), direction-verify (a claim attributed to someone must cite a message where they actually said it), keyword-skip, and no-nag. Sensitive classes layer on extra gates — no-cache for partner content, exact-numeric-fidelity for medical — and route to a human approval queue rather than auto-committing.

Skills as apps

An app reads the graph, writes back, and acts

A skill is an app on the OS. It reads the knowledge graph (silver), and it re-reads its own prior gold output — your edits are preference — so the loop closes. It writes back to gold, sends to the outside world, messages other apps, and improves its own logic from the inside. 23 of the 42 apps run on a uniform chassis, so a new one inherits all of this.

Inside the app: the 7-stage run

click a stage

A deterministic walker enforces every declared substep — the model can't silently skip one.

How apps coordinate — a 3-kind message bus

Single-writer-per-zone would be a straitjacket without cooperation. Apps never touch each other's files; they message over a three-kind bus, and the recipient fulfills on its next run by writing its own zone.

Command

Directed, expires

Point-to-point. task.add, draft.compose_request. Dead-letters if stale.

Event

Broadcast, permanent

"What happened." task.completed. Anyone subscribes; nothing is consumed.

State assertion

Latest-wins

"What's true now." household.wip_count. A newer assertion replaces the old.

The bus is for app-to-app coordination. Routing a task.add to the right owner is worked end-to-end in Reliability.

A separate path: outbound to a human

Reaching a person is its own concern, not just another bus message. Any app that needs to send hands off to /draft (the hub) — it owns voice-matching, the per-body approval gate, and send-verification. The request rides the command bus, but the human-gated send is deliberately distinct from skill-to-skill chatter.

Building & grounding apps

How a new app gets made — and stays honest

Apps aren't hand-coded ad hoc. A new one goes through /forge, an authoring factory with three gated stages — and its design must be grounded in a research doc, which the system then watches for drift.

1 · SCOPE

Dedup → classify → research-gate

Grep the fleet for overlap, classify the tier + shape + bus role, and fire /research if it's a genuinely new domain with no methodology doc yet. One question. Stops you building a skill that already exists — or one with no grounding.

2 · DESIGN

One Opus pass

Reads the foundation docs and the cited research, emits the spec + a schema-validated skill.yml manifest, marks open assumptions. One question.

3 · MATERIALIZE

5-gate validate + atomic promote

Writes files, runs spec-lint + chassis-shape + a live walk on real data, then promotes only if every gate passes.

The three-layer split it enforces

Every app separates spec (abstract logic, tokens only — public-flippable, leaks nothing) from config (your identity: names, paths, MCP tool IDs — gitignored) from the KG (the world facts, in the vault). Editing the wrong layer is the classic leak; the lint blocks literal names or paths in spec text at write-time.

Grounded in research docs — and the system knows when they rot

A genuinely new domain has to cite a methodology doc in gold/Research/ before the skill ships. Each app then declares which docs it depends on and how; a reconciler builds the dependency graph; an audit catches docs that drifted (the field moved under them) or orphaned (nothing uses them).

/forms — fill any web or PDF form from a remembered store, then halt at submit

the 20-page form you don't fill out

It already knows the answers

The N-400 citizenship application runs ~20 pages and ~490 fields — five years of addresses, every employer, every trip out of the country, a wall of biographic detail. The tedium isn't the typing; it's that the answers are scattered across your life.

But the OS already holds almost all of it: identity facts live in the KG; the travel, residence, and employment histories were assembled once by /n400-research (a sibling app whose whole job is reconstructing those five-year timelines) and now persist for reuse; prior forms saved their answers. So /forms maps each field to a value it already has, pulls secrets from 1Password only at fill-time, and finds the genuine handful it doesn't know — then asks for those in one batched question, fills every page, and stops at the submit button. You don't fill the form; you confirm it.

Under the hood it builds a per-edition Form Schema IR (every field → a store_path + validation + an optional op:// secret reference + stamp coordinates), runs a whole-form gap + capacity scan, and never submits or enters payment. New answers project back to the KG through the review queue.

▸3a · Process Analyze the form → extract the Schema IR code LLM Opus

AcroForm PDFs extract deterministically; web forms are an agentic DOM read; XFA auto-seeds stamp coordinates then a model reviews. A schema with zero conditionals is unvalidated → fill blocked.

▸3b · Process Resolve from memory, whole-form scan code

Each field resolves from the KG / prior forms / history tables; secrets at fill-time only. fit_check + a rule-gate run deterministically — "ready" means zero rule violations, recomputed from the rendered form every run, never trusting a prior "done".

▸4 · Synthesize One question for the gaps, then a masked dry-run LLM Opus

Every still-unknown field batched into a single question. Then it renders every page, verifies each value by its printed label (XFA field names lie), and shows a masked dry-run (•••1234) as the sole review gate.

▸5 · Output Fill, store, write back — never overwrite code

Stamps and halts at submit. Collision-free filename + complete .values.json sidecar. Non-secret answers project to the KG at review tier; secrets become 1Password items with only the op:// ref.

/shopping — deep purchase research that compounds its own priors

the fire-table OEM-twin saga

How being forced to check Costco saved $282

The stated want was a 36" round propane fire table at ~$1,299. The app first found the decisive attribute wasn't BTU or diameter — it was table height + form factor — then hunted the OEM-rebrand twin: the identical physical product sells as Lakeview Elysian, Real Flame Aegean, and Costco's "Monaco."

An early narrow sweep skipped warehouse clubs and surfaced ~$1,232 as the floor. But the spec makes the first pass structurally exhaustive — skipping a declared channel is a violation — and Costco membership was already in _Benefits.md. Forced to price the "Monaco": $949.99, 4.9★, same SKU, $282 below the next channel. Lesson, now baked into the spec: breadth before depth, and confirm the exact SKU on the retailer's own page (the aggregator prices kept pointing at the wrong variant).

It bootstraps from a persisted substrate of learned decision criteria + benefits, fans out parallel subagents across every declared channel in one pass, validates specs against primary sources, then stages the cart to Buy and halts.

▸1 · Bootstrap Read the substrate; arm the plan-gate code

Reads five sidecars and writes a run marker that arms a PreToolUse gate: no channel sweep can fire until a plan is approved.

▸3a · Process Decisive attribute + strategy brief LLM Opus

A Kano-style pass names the one attribute that decides success and the binding constraint; approve_plan() unblocks the fan-out.

▸3b · Process Exhaustive parallel channel sweep code LLM Sonnet

One subagent per declared channel — warehouse clubs and the used market included — in parallel. Spec validation is adversarial against primary sources; price history is a deterministic classification.

▸5 · Output Decision matrix; learn the attribute code LLM

The weighted sum + TCO is deterministic; the model picks the best pay-with card. Stage M proposes the new decisive attribute for next time and disarms the gate.

Exhaustive fan-out on run 1

"A first pass that skips a declared channel is a spec violation; it's how the cheapest option gets missed."

The approved-plan gate

A PreToolUse hook blocks any sweep until a plan names the decisive attribute, the binding constraint, and the channel list. Stdlib-only, fail-safe.

/friends — a friend-graph that enforces contact cadence

Names are pseudonyms; the mechanics — Dunbar thresholds, cadence math, schema, rails — are real.

the friendships that quietly lapse

Catching the people you'd hate to lose

The friends you'd drop everything for are often the ones you haven't messaged in months — not from neglect, just life. The drift is invisible until a birthday or a wedding makes it loud, and by then you've lost the thread.

This app keeps the drift visible while it's still small. It learns who matters most and what "in touch" honestly means for each of them — the few you want to talk to most weeks, and the wider circle you'd happily see a couple of times a year — and quietly surfaces the one or two friendships starting to slip, early enough that reaching out still feels natural rather than overdue.

It maintains a graph of friends across Dunbar's four layers (5/15/50/150 → weekly/monthly/quarterly/annual cadence), mines messages for relationship themes (never content), and syncs employers from LinkedIn. Overdue fires at cadence × 1.5.

▸2 · Input Fan out across messages + calendar + chat-silver code LLM Sonnet

A subagent fans out across iMessage/WhatsApp, a 90-day calendar window, and structured chat facts — raw bodies never crowd the parent context.

▸3a · Process Cadence + Dunbar-overdue code

Pure Python: per-friend median cadence, then the threshold. Emits overdue + layer-mismatch candidates but never auto-relabels a layer.

▸3b · Process Theme extraction (shape, not content) LLM Opus

Regenerates each friend's ## Recent Activity with relationship-shape themes from your own outbound patterns — never the friend's words.

▸5 · Output Paraphrase audit → candidate drafts code LLM

A regex audit hard-stops if any friend content would be quoted. Surviving outreach emits draft.compose_request, gated so the reach-out stays invitational rather than performative.

/events — the dates you'd be mortified to forget

Names pseudonymized; memorials are real in the system but shown here only as mechanics.

birthdays, anniversaries, the ones that sting to miss

It knows the date and why it matters

A birthday you forget is rarely about the date; it's that the date isn't anywhere you'll see it in time. The graph already holds them: birthdays, wedding anniversaries, memorials, each attached to a person and how you're connected.

So /events keeps a live record of every acknowledgement-worthy date, watches the calendar against it, and surfaces the next ones early enough to act. A close friend's birthday this week becomes a drafted message you approve; a memorial surfaces too, but never auto-composed — some things you write yourself.

It recomputes from per-person frontmatter (the canonical dates), overlays federal + state holidays and each kid's school no-school days, and routes imminent high-tier dates into the outbound hub. Memorial handling is the privacy-critical path: forced to manual review, never a template.

▸2 · Input Read the canonical dates; sweep only the gaps code Sonnet

Grep per-person profiles for birthday / wedding_anniversary / death_anniversary first — those are the source of truth. A parallel cached-bronze sweep (GCal birthday events + a "happy birthday" scan across allowed group chats) only proposes candidates for people with no date yet; sensitive groups are skip-listed.

▸3 · Process Cadence + holiday + school overlay code

Pure Python: per-date tier-appropriate depth, then overlay federal + WA holidays (via the holidays library) and each kid's no_school_days read across from /school. No model — it's date arithmetic.

▸4 · Synthesize Imminent high-tier → a draft; memorials → manual code LLM Sonnet

Tier-1 dates inside the window become synthetic compose-triggers handed to /draft; a deterministic gate forces every memorial to manual-only. The occasion framing is judgment; the memorial gate is code.

▸5 · Output Regenerate the dashboard; emit the rows code

Rebuilds the next-90-days Events dashboard and emits task.add rows (acknowledgement-due, trip-planning window, childcare gap). Every send still clears a per-body approval gate.

Deterministic by default

Where the model thinks, where code decides — and who's allowed to act

Reliability comes from two disciplines: inside an app, ~80% of substeps are deterministic Python and the model is reserved for judgment; around every run, a hook harness enforces the rules so nothing depends on the model "remembering." The governing rule: a deterministic helper may filter, gather evidence, and gate — but never render a fuzzy verdict.

code deterministic Python
LLM model judgment
tier: Haiku Sonnet Opus

The tier isn't picked by vibes. A central dispatch policy binds each sub-step to a model by the work it does — mechanical pulls and instruction-following run on Sonnet; first-pass synthesis and privacy-sensitive judgment escalate to Opus — and two audit rules flag any skill whose declared tier drifts from the policy. Cost and latency track the difficulty of the substep, not the app.

Judgment code can't fake

In /shopping, the model is handed reviews, spec sheets, the user's real constraint, and a forum's physics — and has to name the one attribute that decides the purchase. For an outdoor TV it concluded the decisive spec is screen reflection, not brightness. No rule set produces that — it's synthesis across messy sources into a single load-bearing call LLM Opus, written back as a durable prior so the next run starts there.

"TV — bright room / outdoor → decisive: Reflection (true-matte / strong AR), NOT brightness. A bright sky mirrors off a glossy screen; brightness can't beat a mirror."gold/Shopping/_DecisiveAttributes.md (a learned, durable prior)

One command, end to end: task.add

Watch a single bus command flow through the boundary twice — first to decide who owns it (model verdict on a deterministic shortlist), then to decide how autonomously it runs (a policy-capped level that may itself use the model).

Step 1 — route to the right owner

A deterministic prefilter resolves the obvious cases; only the ambiguous residue reaches the model; anything still unclear defaults to the household catch-all.

Why the asymmetry

Deterministic-doing-fuzzy fails silently — a confident-looking wrong answer, and keyword sets become whack-a-mole. Model-doing-mechanical merely wastes tokens. So the dangerous mismatch is always the first kind. Arithmetic, dates, lookups, fail-safe gates → code; intent, paraphrase, cross-channel meaning → model.

Step 2 — decide how autonomously it runs

Once a task (or a drafted message) has an owner, an autonomy level governs whether the system acts on its own. The level is per-action — never pinned to a channel — and the classification into a level can itself be a model call, capped by per-skill policy. Automation lives here, and so does the safety.

Automation with a brake

L1 is the default, and L2 must be earned by demonstrating a stable template, idempotency (a re-run supersedes its prior dated writes instead of duplicating them), and low blast radius. The model helps decide the level (is this high-stakes? is the voice profile confident?), but a deterministic policy sets the ceiling — and external sends always verify before claiming "sent." Automation is aggressive where it's safe, gated where it isn't.

Closing the loop on evidence

Autonomy runs the other direction too: an app can notice a task is already done and close it. When a task is created, a deterministic matcher binds it to the thread most likely to carry its done-evidence — an exact participant match plus keyword overlap plus post-creation activity makes a confident bind; fuzzier signals are confirmed before they stick. A per-run detector then watches the bound thread, and new outbound activity after the task was created is the strong "you did it" signal code. It never completes anything silently. It raises a confirmation and you render the Done verdict, the same code-gathers / model-or-human-decides seam as everywhere else. Four apps — /household, /job-search, /health, /forms — close tasks this way off email today; the completion seam is source-agnostic, so the same detector extends to chat without re-architecting it.

The outer layer: a deterministic hook harness

Around every run, 36 hooks fire across five events so the rules never depend on the model remembering them. The PreToolUse zone-guard blocks any write outside an app's owns_zone — the structural fix for one app overwriting another's files; a send-gate resolves the recipient deterministically and hard-blocks any external send until it's vetted and verified; a Stop hook commits all three repos with cross-repo rollback trailers. Over longer horizons, five meta-checks keep drift honest: lint (write-time) · audit (weekly, 75 rules) · review (quarterly) · sweep (mechanical) · health-check (runtime).
